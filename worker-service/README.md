# worker-service

Сервис-исполнитель нейросетевого генератора анимаций для компьютерных игр.
Получает задачи на генерацию, запускает соответствующие AI-модели и
возвращает результаты. Является частью более крупной системы, описанной в
[корневом README](../README.md) и в ВКР, но также поддерживает полностью
автономный режим работы.

## Что умеет исполнитель

Исполнитель поддерживает шесть типов задач, из которых собирается полный
пайплайн генерации анимации по единственному изображению персонажа:

| Тип задачи | Назначение |
| --- | --- |
| `add` | добавляет объект с дополнительного изображения на базовое |
| `delete` | удаляет объект с базового изображения по текстовому описанию |
| `stand` | устанавливает персонажа в позу, заданную скелетом и описанием |
| `animate` | анимирует персонажа по первому кадру и текстовому описанию |
| `rmbg` | удаляет фон с уже сгенерированной анимации |
| `norm_map` | строит карту нормалей для каждого кадра анимации |

Задачи `add`, `delete`, `stand` решаются моделью Qwen-Image-Edit-2509, задача
`animate` — моделью Wan2.2, постобработка (`rmbg`, `norm_map`) — отдельными
специализированными моделями InspyreNet и StableNormal. Для зацикленных
анимаций используется собственный алгоритм поиска цикла на основе
DINOv2-эмбеддингов кадров.

## Режимы работы

Исполнитель может быть сконфигурирован для работы в двух режимах: одиночное
приложение, выполняющее заданный порядок операций локально, и сервис,
принимающий задачи на выполнение и возвращающий результаты обратно через
брокер сообщений и объектное хранилище.

В обоих случаях конфигурация производится посредством YAML-файлов. Этот
формат является человекочитаемым, удобен для ручного редактирования и при
этом хорошо поддерживается средствами программной обработки.

### Одиночное приложение

Основной способ взаимодействия с разработанным решением в рамках работы.
Генерация выполняется на одном устройстве, взаимодействие происходит
непосредственно с исполнителем без сервис-менеджера, брокера сообщений и
объектного хранилища.

Установка зависимостей и запуск:

```sh
uv sync --extra standalone
worker-standalone -c config/config.yml config/pipeline.looped.yml \
    -- --inputs.first_frame=assets/inputs/girl.png
```

Здесь `config/config.yml` описывает общие параметры — путь до моделей,
ограничение по видеопамяти, директорию для сохранения результатов и политики
повторов. `config/pipeline.looped.yml` описывает сам пайплайн: входы, набор
задач и зависимости между ними. После `--` идут пары `--inputs.<имя>=<путь>`,
переопределяющие значения входов пайплайна.

Результат работы задачи с идентификатором `task_id` сохраняется по пути
`<outputs.base_dir>/<task_id>/<n>/<output_name>.png`, где `n` — номер
результата или кадра. Если идентификатор задачи не задан, автоматически
используется последовательное значение `anonymous_{i}`.

### Сервис

Режим, в котором исполнитель выступает частью более крупной системы. Задачи
поступают последовательно из соответствующих топиков брокера сообщений и
могут быть прерваны при помощи механизма отмены. Артефакты передаются через
S3-совместимое объектное хранилище.

```sh
uv sync --extra service
worker-service -c config/service.yml
```

В конфигурации сервиса доступны:

- `tasks[].consumer.topics` — список топиков, из которых данный исполнитель
  потребляет задачи. Меняя это значение, администратор может настраивать
  конкретного исполнителя на обработку только определённых видов задач:
  один — на долгие задачи генерации анимации, другой — на более быстрые
  операции работы с изображениями и постобработку;
- `storage` — параметры S3-совместимого хранилища (бакет, префикс, регион,
  endpoint, ключи доступа);
- `cancellation` — каналы отмены: подписка на топик Kafka и/или
  HTTP-эндпоинт менеджера для проверки статуса задачи непосредственно перед
  её выполнением;
- `logs` — экспорт логов в Loki;
- `metrics` — экспорт метрик в Prometheus на указанном порту.

Полная схема конфигурационных файлов исполнителя приведена в приложении А
к ВКР.

## Конфигурация пайплайна

Конфигурация пайплайна описывает справочную информацию (имя, версия), входы,
набор задач и зависимости между ними. Каждая задача из списка содержит
информацию о своём типе и входных артефактах, а также может иметь
опциональный идентификатор для ссылок из других задач.

Пример пайплайна полной генерации анимации одного персонажа по единственному
изображению:

```yaml
version: v1
name: Генерация анимации
inputs:
  character: assets/inputs/character.png

tasks:
  - id: pre
    name: Удаляем лишние объекты
    type: delete
    inputs:
      character: pipeline.inputs.character
      prompt: "Remove items from the character's hands"

  - id: animate
    name: Генерируем анимацию
    type: animate
    inputs:
      first_frame: pre.outputs.character
      prompt: "A character running in place"

  - id: output
    name: Удаляем задний фон
    type: rmbg
    inputs:
      frames: animate.outputs.frames

  - id: output_norm
    name: Создаём карту нормалей
    type: norm_map
    inputs:
      frames: animate.outputs.frames
```

Перед выполнением пайплайна на его основе конструируется ориентированный
ацикличный граф зависимостей. На этом этапе система выполняет валидацию и
сообщает об отсутствии обязательных значений, некорректных ссылках или
циклических зависимостях.

Полная схема конфигурации пайплайна приведена в приложении Б к ВКР, готовые
примеры — в `config/pipeline.yml` и `config/pipeline.looped.yml`.

## Структура исходного кода

Сервис разработан на Python 3.10+ с применением подхода Domain-Driven Design.
Прикладная логика отделена от инфраструктурных деталей: каждый режим работы
состоит из общей доменной логики, взаимодействующей с внешним миром через
комбинацию доступных реализаций внутренних интерфейсов.

```
worker-service/
├── src/
│   ├── cli/             # точки входа standalone и service
│   ├── config/          # парсинг и валидация YAML-конфигов
│   ├── controller/      # адаптеры для Kafka и сигналов отмены
│   ├── domain/          # бизнес-логика
│   │   ├── pipeline/    # реализации типов задач, поиск цикла
│   │   ├── model/       # доменные модели: персонаж, кадры, цикл
│   │   └── worker/      # device-менеджер, scheduler, Worker
│   ├── infra/           # интеграции: Kafka, S3, Loki, Prometheus, лог
│   ├── repository/      # доступ к локальному и S3-хранилищам
│   ├── app/             # порты доменной логики и реализации задач
│   ├── metrics/         # сбор и экспорт метрик
│   ├── Service.py       # сборка сервисного режима
│   └── Standalone.py    # сборка standalone-режима
├── config/              # примеры конфигурационных файлов
├── tests/               # юнит-тесты на pytest
├── standalone.py        # точка входа в standalone-режиме
├── service.py           # точка входа в сервисном режиме
├── Dockerfile.standalone
└── Dockerfile.service
```

## Менеджер моделей

Так как сервис обслуживает несколько типов задач, он должен использовать
несколько моделей. Одновременная загрузка всех моделей в видеопамять часто
невозможна, особенно при работе на потребительских видеокартах. Поэтому в
исполнителе реализован менеджер ресурсов, относящий каждую модель к одному
из трёх состояний: активная, неактивная (загруженная в память, но не
используемая в текущей задаче) и выгруженная.

При нехватке памяти политика выселения решает, какая модель должна быть
выгружена, чтобы освободить ресурсы под текущую задачу. По умолчанию
используется LRU-стратегия, выбирающая модель, которая не использовалась
дольше всего. Архитектура механизма выселения позволяет улучшать политики
без переписывания остальной системы.

## Запуск в Docker

Для каждого режима предусмотрен отдельный `Dockerfile`. Сборка
standalone-образа:

```sh
docker build -f Dockerfile.standalone -t worker:standalone .

docker run --rm --gpus all \
    -v $PWD/config:/app/config:ro \
    -v $PWD/models:/app/models \
    -v $PWD/assets:/app/assets \
    worker:standalone \
    -c /app/config/config.yml /app/config/pipeline.yml
```

Сборка сервисного образа требует доступа к sibling-репозиторию `proto/`,
который подключается как именованный build-контекст:

```sh
docker build -f Dockerfile.service \
    --build-context proto=../proto \
    -t worker:service .

docker run --rm --gpus all --network host \
    -v $PWD/config:/app/config:ro \
    -v $PWD/models:/app/models \
    worker:service \
    -c /app/config/service.yml
```

Внутри контейнера пути в YAML-конфигурациях должны быть абсолютными
(например, `/app/models`), поскольку конфиг резолвит относительные пути от
расположения console-скрипта, а не от текущей рабочей директории.

## Требования к окружению

Для воспроизведения результатов работы рекомендуется конфигурация,
сопоставимая с использованной в ВКР: видеокарта уровня NVIDIA A100 / RTX
4090 / RTX 5090 с не менее чем 24 ГБ видеопамяти. На менее мощных
устройствах возможен запуск с уменьшенным числом кадров и более агрессивной
политикой выселения, но качество и время генерации могут пострадать.

При работе с моделями требуется наличие `huggingface-hub`-токена с доступом
к используемым весам. Каталог с уже загруженными моделями указывается в
конфиге через поле `models.dir`.
